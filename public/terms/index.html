3. Entering the Alpha

Checking the in-game box and selecting Continue to sign-in confirms that you understand this notice and agree to these Alpha Terms for that entry attempt. The checkbox begins unchecked each time and its UI state exists only in browser component memory. If an admitted player completes authentication, Warpkeep stores private, immutable evidence containing the player's FID, this Terms version, and the acceptance time. That evidence enforces the versioned entry gate and supports security and audit; it is not public game state and does not contain the checkbox state, QR payload, proof, signature, access token, or session cookie.

Use Warpkeep only if you can legally accept these terms. Use only an account you are authorized to control, follow applicable law, and do not disrupt the service, bypass access controls, impersonate others, automate abusive traffic, or access another player's non-public data.

4. Accounts and transactions

The current entry flow uses Farcaster authorization. Warpkeep does not need your seed phrase or private key; never send either to the project. Warpkeep does not ask the browser to connect a wallet, sign a wallet message, approve a token, submit a transaction, or make a payment. Any SNAP burn is a separate user-initiated ordinary token burn on Ethereum mainnet. Alpha access does not establish ownership of a castle, token, account, or future benefit outside the current experimental game state.

5. Privacy and public game state

The Alpha Privacy Notice explains the FID-only server identity, session cookie, short-lived access token, public SpacetimeDB game projections, retention limits, service providers, and available privacy choices. Some admitted-player state, including FID, trusted public Farcaster presentation, castle state, and—after entry—aggregate SNAP-burn and Mark statistics, is public to connected clients by design. Wallet addresses and individual burn-event receipts are not public game state.
