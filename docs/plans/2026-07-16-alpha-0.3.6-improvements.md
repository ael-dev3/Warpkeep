# Warpkeep Alpha 0.3.6 improvement workstream

**Status:** Active Alpha 0.3.6 candidate workstream in draft PR #44, based
directly on `main` after the Alpha 0.3.5 integration and release-truth work
landed through PRs #40 and #43. The first player-facing work item is now
implemented in source and advances the checked-in candidate version to 0.3.6;
it is not merged, deployed, tagged, or verified as a public release. Alpha
0.3.5 remains the verified public release.

## Intent

Alpha 0.3.6 collects focused, reviewable improvements that build on the
GameReady castle, authored landscape-base, Realm identity, authentication, and
Pages foundations already present in Alpha 0.3.5. Each implementation should
state its player-visible outcome, preserve existing authority boundaries, and
arrive with validation proportional to its risk.

## Intake tracks

Candidate work belongs in one of these tracks before implementation begins:

1. **Realm presentation and interaction** — lighting, palette, terrain and
   castle readability, camera behavior, selection, labels, responsive layout,
   accessibility, and measured rendering performance.
2. **Reliability and recovery** — explicit failure behavior, loading and
   fallback states, deterministic cleanup, asset integrity, observability, and
   reproducible QA.
3. **Identity and admission UX** — clearer Farcaster, session, admission, and
   profile presentation without weakening server authority or inventing client
   trust.
4. **Security and protocol assurance** — narrowly scoped hardening that keeps
   Worker, OIDC, SpacetimeDB, wallet, Marks, and production-data boundaries
   explicit.
5. **Release quality** — in-menu patch notes, dated release truth, provenance,
   reconstruction records, mobile/desktop browser coverage, and deployment
   evidence.

The accepted candidate scope is now the Realm readability and stability work
item below. Any additional work must still be added as a named checklist item
with acceptance criteria before code or asset changes are made; this candidate
must not become an unbounded polish branch.

## Accepted work item — Realm readability and stability

Detailed evidence, root causes, and acceptance criteria live in
[`2026-07-17-realm-render-stability.md`](2026-07-17-realm-render-stability.md).
The bounded implementation is:

- [x] apply one idempotent role-aware diffuse calibration across all castle
  LODs and a smaller calibration to authored landscape bases, without changing
  GLB bytes, textures, transforms, authored roughness/metallic/normal response,
  or provenance;
- [x] redirect the existing neutral fill toward camera-facing masonry and
  reduce the amethyst fill without raising global exposure, terrain energy,
  light count, shadow allocation, passes, or animation demand;
- [x] widen local terrain foundation/blend support beyond the authored island
  envelope at all canonical castle slots, while preserving one exact shared
  castle/base transform and every authoritative coordinate;
- [x] suppress depth-tested terrain-cell outlines on occupied castle cells so
  interaction feedback does not visually cross the wider base;
- [x] preserve one permanent direct identity rail for every projection-visible
  founded castle, with no camera-distance membership, cluster, overflow, or
  displacement path; allow overlap before losing spatial truth;
- [x] keep ordinary wheel/pinch input above a readable zoom floor while the
  explicit Realm overview fits the actual convex rendered-terrain perimeter
  with a conservative raised-scene margin so canonical slots remain
  inspectable;
- [x] advance package/lockfile version and synchronize the 0.3.6 in-menu patch
  chronicle, changelog, README, release record, roadmap, and agent rules as
  candidate truth; and
- [ ] complete the full automated, production-build, rendered-WebGL, and
  responsive browser matrix recorded in the detailed work item before review.

No accepted item changes authentication, admission, backend protocol,
authoritative world or castle state, production data, wallet, Marks, Worker,
SpacetimeDB, DNS, or deployment state.

## Development guardrails

- Keep this pull request in draft while requirements are still being collected.
- Keep this branch based on `main`; do not merge or deploy it as an incidental
  setup step.
- Do not change DNS, production services, admission lists, wallet state,
  SpacetimeDB data, Marks, or other external authority without an explicit task
  that places that mutation in scope.
- Preserve exact runtime-asset hashes, provenance classifications, placement
  contracts, and license boundaries unless a separately authorized replacement
  is recorded.
- Separate browser presentation from authoritative game state. A visual or UX
  improvement must not silently create a new gameplay field, action, or trust
  decision.
- Keep changes reviewable. Prefer cohesive commits and record the validation
  relevant to each change rather than deferring all evidence to the end.
- When the first player-facing 0.3.6 change is accepted, advance the package
  version and add the corresponding `0.3.6` in-menu patch-note entry, changelog
  section, and dated release record in the same workstream. Thereafter every
  player-facing change must keep those surfaces synchronized.

## Work item template

For each accepted improvement, add:

- **Outcome:** what becomes materially better for the player or operator;
- **Scope:** the exact code, asset, documentation, or service boundary involved;
- **Non-goals:** nearby authority or behavior that remains unchanged;
- **Acceptance criteria:** observable conditions required for completion;
- **Validation:** unit, integration, rendered-browser, security, performance, or
  deployment evidence required before review;
- **Release note:** the concise player-facing statement, or an explicit reason
  the change is internal only.

## Baseline validation

The workstream begins from `main` commit
`bd957556169b979fd9853790062d45f1c0e29bf5`, after Alpha 0.3.5 integration and
release-truth finalization. New commits must not rely on that historical result
when they alter the affected surface; rerun the relevant checks and update the
PR description with current evidence.

At minimum, a release-ready 0.3.6 candidate should have:

- a clean TypeScript build and complete automated test suite;
- exact runtime-asset and file-size verification;
- production-output exclusion checks;
- rendered browser coverage for every changed player-visible path and supported
  responsive class;
- updated security, provenance, reconstruction, and patch-note records wherever
  the implementation changes those contracts; and
- protected deployment evidence before any candidate is described as live.
