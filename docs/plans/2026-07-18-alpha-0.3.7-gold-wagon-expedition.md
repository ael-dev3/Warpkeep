# Alpha 0.3.7 Gold wagon expedition pilot

**Status: implementation candidate, stacked on the Alpha 0.3.7 resource
authority. This plan authorizes source implementation and a draft review PR
only. It does not authorize a production SpacetimeDB publication, genesis
seed mutation, founder backfill, Pages deployment, tag, release, or merge.**

## Product boundary

Genesis 001 remains its existing 1,261-cell world with 250
`resource-capable` anchors. The first Gold loop is a deterministic, reviewable
pilot of **24 Tier-I Gold Mine sites** drawn only from those already-canonical,
passable anchors. It is not a substitute for the separately proposed
10,000-cell / 2,000-anchor expansion, and it must never describe itself as one.

Each founder may dispatch one Hegemony supply wagon from their founded keep to
an unoccupied Gold Mine. After a server-derived outbound travel interval, the
wagon gathers **one whole Gold per completed server minute** for 30 days. At
the end of gathering it begins a server-derived return interval, releases the
site for another wagon, and becomes available to its originating keep only on
return. A full uninterrupted 30-day gathering phase is 43,200 Gold before the
existing account cap. The operator, not the browser, owns all timestamps,
rates, routes, limits, and settlement.

Gold terrain yield from the preceding resource-authority candidate is set to
zero for this pilot. Food, Wood, and Stone retain their compiled terrain policy.
This gives Gold a single authoritative issuance path and avoids a hidden second
production source.

## Authoritative state and privacy

The additive server schema separates a public map projection from private
economy state:

- The public Gold-site catalog exposes only a stable site id, canonical map
  coordinate, tier, and active status.
- Public occupancy exposes a site id, originating castle id, lifecycle phase,
  and server-derived lifecycle timestamps. It lets every player see that a mine
  is occupied without exposing an owner FID, private balance, accumulated
  production, request key, or account row.
- Private expedition and idempotency records bind an admitted caller to their
  one current origin castle and request. They store the settlement cursor and
  any private credited amount.
- A one-shot internal schedule resolves arrival, end-of-gathering, and return.
  It validates its expected lifecycle revision, settles only whole elapsed
  minutes at the compiled policy rate, and is safe to replay or arrive stale.
  It must never trust a client timer or be callable as a player action.

Dispatch derives the caller, founder admission, terms acceptance, castle,
canonical site, eligibility, route, timestamps, rate, account, and maximum
concurrency on the server. Its client request id is used only for idempotency.
It atomically rejects an occupied site, a second active wagon for the same
castle, malformed or stale graph state, and an invalid map placement. Repeated
or concurrent requests cannot create two occupancy leases or double-credit
Gold. Collection and finalisation use the same private settlement cursor, so
they cannot credit a minute twice.

## Assets and presentation

The pilot uses the complete reviewed asset family rather than a proxy-only
mockup:

- High, Balanced, and Compact Hegemony Gold Mine GLBs become integrity-pinned
  runtime LODs only after their recorded atlas metadata and provenance checks
  pass. The original reference candidates remain outside `public/`.
- The provenance-pinned transparent Gold Mine record illustration is the
  inspection-card art. It remains decorative and pointer-inert; live values
  are sourced from the authenticated authoritative projection.
- The Hegemony NoTelescope GameReady supply wagon is the canonical source for
  reviewed High, Balanced, and Compact runtime outputs. Its skinned movement
  animations are shared and used only inside the defined nearby budget;
  distant wagons use the compact representation or a non-animated marker.
- Existing immutable Food, Wood, Stone, Gold, and Marks icons remain HUD-only
  visual vocabulary. They do not create balances or currency authority.

The Realm renders only validated public site and occupancy data. It raycasts a
Gold Mine as a separate target, opens an accessible inspection record, and
shows a clear occupied/available state. The renderer interpolates only between
server-derived lifecycle points; it sends no motion, clock, route, production,
or occupancy writes. If the authoritative projection is incomplete or invalid,
the safe presentation is no node rather than a fabricated local mine.

## Review and rollout proof

The source candidate needs deterministic policy, reducer, schema, migration,
binding, asset-integrity, presentation, accessibility, and browser-render
coverage. Tests must prove exactly 24 valid unique sites; no site on an
impassable/non-resource-capable cell; public/private subscription boundaries;
idempotent same-request dispatch; contested dispatch rejection; one active
wagon per castle; stale schedule no-ops; whole-minute settlement; cap behavior;
and no duplicate Gold from passive terrain.

Before any future owner-approved production release, publish the additive
module without deletion, run only a specifically approved guarded backfill or
site installation operation, verify aggregate invariants without identifying
players, deploy the exact reviewed Pages SHA, and perform bounded live checks.
No test result, review approval, merge, or browser build grants that production
authority by itself.
