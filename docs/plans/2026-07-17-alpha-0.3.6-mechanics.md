# Alpha 0.3.6 Mechanics Workstream

**Status:** deferred after the verified Alpha 0.3.6 release; no gameplay
mechanic was implemented or activated.

**Release intent:** retain a separately reviewable mechanics proposal for a
future version only after its rules, authority, recovery, and release truth are
complete. It cannot be retrofitted into the immutable Alpha 0.3.6 release.

## Initial direction

This workstream follows the published next gameplay slice: one small,
server-authoritative loop for resource production and deterministic construction
queues.

It is deliberately narrower than combat, trading, wallet activity, alliances,
or an economy simulation. A player should be able to understand what their keep
produces, choose one bounded construction intent, and see the authoritative
result after a reload or on another client.

## Prepared presentation asset

The dated records under `docs/reference/resources/` retain transparent,
integrity-pinned reference masters for gold, food, stone, and wood. They are
deliberately outside the Pages `public/` tree: this planning slice does not
bind any image to a balance, cost, reward, currency, or playable mechanic.
Their sources, alpha-cleanup process, and limited-use provenance remain beside
the exact masters.

## Non-negotiable boundaries

- The browser may request an intent and present a public projection; it never
  owns resource balances, timers, ownership, queue ordering, or completion.
- Resource production derives from authoritative terrain and server time, never
  from client clocks, Farcaster profile data, local storage, or rendering state.
- Construction acceptance, cost, queue order, completion, and idempotency must
  be deterministic and atomically resolved by the server.
- FID remains identity. Handles, avatars, and social activity remain display
  metadata and do not grant resources or construction authority.
- Marks, wallet transfers, promised rewards, admissions, combat, alliances,
  and private accounting are outside this PR unless separately authorized.

## Decisions required before implementation

1. Define the smallest initial resource set and its terrain-derived production
   formula, including rounding and time boundaries.
2. Define one construction catalog with explicit prerequisites, cost, duration,
   queue capacity, cancellation, and completion rules.
3. Specify idempotent request keys, conflict behavior, server-time source, and
   recovery after reconnect, restart, duplicate delivery, or stale client data.
4. Define the minimum public read model for a player keep and the privacy rules
   for other keeps.
5. Agree on player-visible copy and exact future release-note language only
   after the implementation is authoritative and testable.

## Intended implementation order

1. Record the approved rules and invariant table before adding schema or UI.
2. Add additive server schema and deterministic reducers with generated-binding
   parity checks.
3. Build isolated tests for production accrual, queue ordering, duplicate
   intents, ownership checks, time boundaries, reload, and concurrent clients.
4. Expose a minimal, sanitized browser projection with safe loading and error
   states; add no optimistic authority.
5. Add one bounded menu/Realm interaction only after the server path is proven.
6. Update the complete player-visible release-truth set, run exact-head
   verification, and obtain explicit approval before any deployment.

## Acceptance gate

This workstream is not ready to merge as a gameplay feature until it proves:

- the same authoritative state on reload and across multiple clients;
- deterministic results for repeated or reordered requests;
- no mutation by an unauthorised FID or stale client projection;
- recovery evidence for interrupted queue processing;
- no fabricated player-facing balances, durations, or rewards; and
- complete tests, generated-binding checks, review, and release notes for the
  exact future candidate head.

Until those gates are met, this PR is a draft review boundary rather than a
claim that resources or construction are playable.
