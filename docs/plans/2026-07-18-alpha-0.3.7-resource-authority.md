# Alpha 0.3.7 resource authority

Status: implementation candidate. This document does not authorize a production
SpacetimeDB publication, data backfill, Pages deployment, tag, or release.

## Release objective

Alpha 0.3.7 makes Food, Wood, Stone, and Gold persistent SpacetimeDB-owned
state for every founded castle. Marks remains governed by the existing private
SNAP-burn ledger and is never converted into, duplicated by, or spent through
the economic-resource authority.

The browser may present only the authenticated player's economic inventory.
Peer inventories remain private. The six-table public Realm subscription stays
unchanged.

## Frozen decisions

- Economic balances are whole unsigned 64-bit units.
- The private `resource_account_v1` row is keyed by FID and uniquely bound to
  one castle in `GENESIS_001`.
- Initial balances are zero Food, Wood, Stone, and Gold. Resource icons and
  counters may therefore present zero without inventing client-only state.
- Player chrome is intentionally compact: a PFP-only upper-left launcher owns
  My Keep, Explore, Settings, and Main Menu; a transparent upper-right rail
  presents Food, Wood, Stone, Gold, and Marks from the authenticated private
  projection. No persistent bottom navigation or expanded identity/keep block
  is part of the player Realm.
- Castle inspection art uses a transparent overhanging stage. Player and
  observer geometry remain scoped independently, and measured collision zones
  reserve only the persistent portrait and resource footprints.
- Production settles in complete 10-minute quanta using only the reducer or
  procedure transaction timestamp. Browser clocks, elapsed-time inputs,
  balances, rates, FIDs, and castle IDs are never accepted as authority.
- Each balance is capped at 1,000,000 units. A collection at the cap still
  advances the settlement cursor, so later spending cannot resurrect elapsed
  production that was deliberately discarded at the cap.
- Terrain yield is a compiled, versioned mapping. Unknown terrain or policy
  versions fail closed.
- `collect_resources_v1` accepts no input. Repeated or concurrent calls cannot
  double-credit because each transaction settles only complete quanta after
  the stored cursor.
- `get_my_resource_state_v1` accepts no FID and returns the caller's stored
  balances, pending yield, private Marks balance, server observation time,
  cursor, revision, terrain, and exact policy versions.
- Resource APIs require a current admitted and bootstrapped player, exact
  ownership, the current Alpha Terms acceptance, and a consistent founder,
  resource, and Mark graph.
- The current backend protocol remains version 3. The resource procedure has
  its own exact versioned wire name and policy version, preserving an additive
  zero-downtime module-first rollout for Alpha 0.3.6 clients. A 0.3.7 client
  still fails closed when the new procedure is unavailable or malformed.

## Additive migration

`resource_account_v1` is appended after every deployed table. No existing
table, field, index, reducer, procedure, or wire shape changes.

New founders receive a resource row atomically. Existing founders require an
idempotent Hermes-only backfill that:

1. validates the complete existing founder graph and an exact expected count;
2. rejects every conflicting resource row before writing;
3. creates only missing rows with compiled starting balances and the server
   timestamp as their first settlement cursor; and
4. becomes a no-op on an exact rerun.

`admin_get_alpha_status_v4` exposes aggregate counts and violation counters
only. It never returns FIDs, identities, balances, profiles, tokens, or logs.

## Production hold and rollout

Local builds, generated bindings, disposable migration fixtures, tests, and a
draft GitHub candidate are authorized implementation work. Production remains
held behind a fresh explicit owner approval. After approval, the safe order is:

1. publish the additive module with deletion disabled;
2. run the guarded founder backfill;
3. verify the counts-only v4 aggregate and a bounded own-account read;
4. deploy the exact reviewed Pages SHA; and
5. verify the exact live build and resource path without mutating balances.

Rollback must never publish an older destructive schema. Pages can roll back
independently while the additive tables and functions remain present.
