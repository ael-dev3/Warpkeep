# Hegemony Gold Mine review candidates

These three exact Gold Mine GLBs are staged for a separate technical and
product review. They are not imported by the application, copied into Pages,
placed on the map, associated with a balance, or evidence that Gold gathering
is playable. The machine-readable [candidate record](manifest.json) and
`npm run verify:gold-mine-candidates` pin their bytes and structural contract.

## Separate 2D inspection art

The owner also authorized a distinct transparent 2D visual for the reviewed
Gold Mine inspection card. Its immutable runtime output, edit prompt, alpha
audit, and authorization boundary are in the [Gold Mine inspection-art
record](record-art/README.md). The draft Gold Wagon integration mounts that
asset only as decorative inspection art for a validated public site; it does
not authorize production seeding, deployment, or browser economic authority.
This does not promote, alter, or deliver any GLB in `runtime-candidates/`.

All profiles are exact, digest-named copies of the supplied runtime inputs.
They are GLB 2.0 files generated by glTF-Transform 4.4.1 with one scene, node,
mesh, primitive, and material; three embedded WebP images; and required
Meshopt, WebP, and quantization extensions. The authored contract records +Y
as up, +Z as front, a footprint-center interaction pivot, and an engine-side
oval/cylinder collider rather than render-mesh collision.

The observed grounded bounds are close to, but not mathematically equal to,
the manifest's rounded ground plane: Blender-space Z is approximately
`0.000187` to `7.934813`. A future renderer must use a reviewed terrain-contact
tolerance rather than assume an exact zero from this candidate record.

## Deliberate review gate

The High candidate's `wk_atlas_size` metadata agrees with its 1024² atlases.
The supplied Balanced and Compact candidates contain 512² and 256² atlases but
both still declare `materials[0].extras.wk_atlas_size: 1024`. This staging PR
preserves all supplied bytes unchanged. A future integration PR needs explicit
approval either to use the true decoded dimensions without that metadata or to
apply and fully verify the bounded metadata-normalization pattern used by the
castle families. It must never silently alter these candidate bytes.

Some authoring extras also contain non-runtime source counts that differ from
the packed geometry. Future code must use the reviewed external candidate
record and its exact hashes, not scene extras, for LOD selection, budgets, or
world authority.

## Promotion prerequisites

1. Land or identify a reviewed Genesis 001 expansion head with exactly 10,000
   authoritative cells and exactly 2,000 resource-capable anchors.
2. Obtain an explicit decision on the Balanced/Compact atlas-metadata mismatch.
3. Review a server-authoritative site/order/account design and deterministic
   placement artifact; Gold remains distinct from Community Marks.
4. Add a separate runtime integration that covers terrain grounding, instanced
   LOD loading, accessibility, state synchronization, security, tests, and
   release/deployment gates.

Until then, keep these files outside `public/`, do not add a frontend loader or
world coordinates, and do not publish, seed, or deploy anything.
