# Alpha 0.2 authentication and release lessons

Alpha 0.2's authentication path failed closed during final production QA. A legitimate Farcaster approval could create a one-time challenge, but the Worker could not complete its private SpacetimeDB authorization-epoch lookup, so it issued no player token and created no gameplay state.

Two distinct runtime defects were involved. First, Workerd rejected `redirect: "error"` before issuing the outbound request. Using `redirect: "manual"` remained fail closed because every redirect surfaced as a non-success response and was rejected. After that repair, production still failed because the resolver stored the runtime `fetch` function and invoked it as `this.fetcher(...)`. That call form supplied the resolver instance as the host function's receiver. The deployed runtime rejected the accidental receiver. Calling through a local binding—`const fetcher = this.fetcher; await fetcher(...)`—restored receiverless invocation.

The security boundary held throughout: browser errors remained generic, no token was minted on resolver failure, the private allowlist stayed private, and the protected aggregate remained 61 world cells with zero admissions, enabled identities, players, or castles.

## 1. Initial architecture and trust boundaries

The intended chain was deliberately layered:

```text
browser action
  -> one-time Worker challenge
  -> Farcaster approval and SIWF proof
  -> independent Worker proof verification
  -> private SpacetimeDB authorization-epoch procedure
  -> bridge-issued ES256 player token
  -> authenticated SpacetimeDB connection
  -> private admission decision
  -> public world/player/castle projections
```

The browser owns presentation, never identity authority. Optional username, display name, avatar, custody address, and verification metadata cannot choose the authoritative FID. The Worker independently verifies the completed proof, consumes the challenge, signs a short-lived Hermes service token, and calls the fixed protected `admin_get_fid_auth_epoch` procedure. SpacetimeDB then independently validates issuer, audience, subject, roles, deadlines, admission, and epoch before gameplay state can exist.

A remembered browser session is only a closed-alpha convenience. It can restore a still-valid signed session, but it cannot bypass the module's admission check.

## 2. Why the production bug was difficult

The outer behavior was intentionally quiet:

- the browser received `503 authorization_unavailable`;
- the Worker emitted the aggregate event `auth_epoch_failed`;
- raw upstream errors were never returned;
- the resolver caught dependency failures, so an unhandled exception count of zero was expected;
- shared state correctly remained unchanged.

That secure result collapsed several hypotheses into one: signing, request construction, DNS/TLS setup, redirect behavior, timeout, upstream rejection, response-stream failure, media type, JSON parsing, and epoch validation. “No exception” did not mean “the resolver worked”; it meant the resolver handled its failure.

The UI added another ambiguity. An intentionally empty admission list produces the same denial panel after a fresh successful exchange and after restoring a remembered valid session. Seeing the correct panel did not prove which network path had run.

## 3. Local Node, local Workerd, and deployed Workerd

Node unit tests were strong for claims, headers, timeouts, body limits, parsing, and failure mapping, but Node accepted request behavior that deployed Workerd rejected. A pinned local Workerd spike exposed the unsupported redirect mode. It then successfully performed a direct signed procedure call, yet production still failed because that spike invoked global `fetch(...)` directly while the real resolver invoked a stored function through an object property.

Runtime parity therefore needs three kinds of sameness:

1. the runtime family and compatibility configuration;
2. the request and upstream contracts;
3. the production abstraction and its exact invocation semantics.

A runtime-exact but call-path-inexact experiment can still miss the bug.

## 4. The redirect-mode incompatibility

The resolver originally used `redirect: "error"`. That looked right for a security-sensitive request, but Workerd rejected the option before sending a subrequest. The repair used `redirect: "manual"`; it did not allow redirects. It made them visible as non-success responses, which the existing `response.ok` guard rejected.

Portability did not require weakening the redirect boundary. It required expressing the same boundary through a supported runtime contract.

## 5. Why generic `auth_epoch_failed` telemetry was insufficient

The generic event proved the exchange had reached the private authorization boundary and failed closed. It could not say whether failure occurred while signing, creating the request, reading the body, timing out, receiving an upstream rejection, or validating the result. Each speculative production repair otherwise consumed another deploy and potentially another human approval.

Logging raw errors was not acceptable: a runtime message might include an endpoint, provider response, header, token, caller detail, or other operational secret. The solution was a closed catalog of static, privacy-safe stages.

## 6. Closed failure stages

The resolver now classifies only these internally constructed stages:

- `signing`
- `fetch_request`
- `fetch_body`
- `timeout`
- `upstream_status`
- `response_validation`

Each stage names a bounded code region rather than caller-controlled text. Public exchanges remain generic. Internal static events map one-to-one, for example `auth_epoch_failed_fetch_request`. Structurally similar objects, mutated stage values, and unexpected bugs cannot fabricate a category.

No stage contains a FID, proof, nonce, request identifier, token, key, URL, upstream body, status detail, or exception string. Separating `fetch_request` from `fetch_body` was decisive: a rejection before a `Response` exists is a different class from failure while consuming an otherwise returned stream.

## 7. The server-only synthetic resolver probe

The permanent probe invokes the same configured resolver as a real exchange while removing browser and human dependencies. It is server-authenticated, accepts no body, query, browser `Origin`, or caller-selected identity, uses one fixed synthetic safe integer, and calls the same read-only epoch procedure.

Success returns only `{ "ok": true }`. A recognized failure returns only `{ "ok": false, "stage": "fetch_request" }`. The response never contains the synthetic argument, epoch, token, endpoint, upstream body, raw status, or error.

Repeated green production probes established signing and the complete private resolver path before another Farcaster approval was requested. Protected aggregate inspection proved the probe created no admission or gameplay state.

## 8. Why temporary `wrangler tail` collectors lost evidence

Real-time tailing was initially treated like an evidence store. It is not one. One collector reduced output line by line, while the Wrangler version in use emitted relevant route data as multiline pretty JSON. The decisive frame was split and never classified. Another temporary observer was no longer alive when its result was needed.

An unobserved approval is neither success nor failure; it is unknown. Real-time logs may be sampled, buffered, formatted differently under a TTY, or dropped. Tail is useful for supervised diagnosis, but durable, bounded Worker events and exact version coordinates are stronger release evidence.

## 9. PTY and multiline observation lessons

The repaired observer used a PTY-compatible session, verified its handshake and target version, parsed multiline frames, checked process liveness, and retained only allowlisted route, status, static event, and exception-count fields. A malicious synthetic-frame test proved sensitive-looking fields could not cross the reducer.

Terminal formatting, buffering, wrapping, and process lifetime are part of an operational evidence contract. Tooling should be tested against what the command actually emits, not what a one-line example suggests.

## 10. Remembered-session contamination

Both of these flows can end at the expected empty-admission UI:

```text
fresh challenge -> proof exchange -> valid token -> admission denial
remembered valid token -> admission recheck -> admission denial
```

Therefore the panel alone could not prove a fresh exchange. A previously used profile also left ambiguity about storage, pending Farcaster requests, tabs, and which challenge a human approval belonged to. Fresh-auth evidence and remembered-session evidence became separate named phases.

## 11. Why a pristine one-window, one-tab profile mattered

The definitive run began with one temporary profile, one window, one tab, the canonical origin, anonymous state, no Warpkeep storage, no pending request, no challenge or exchange before interaction, the expected deployed build, and a live exact-version observer. The profile was removed after fresh and remembered phases.

That made browser state a verified precondition instead of an uncontrolled variable and prevented approval from correlating with the wrong challenge or tab.

## 12. Why Codex initiated **ENTER REALM**

Codex controlled the public website action while the owner retained the only identity-bearing step. It verified the pristine baseline, selected **ENTER REALM** exactly once, observed one challenge, confirmed no exchange preceded approval, and paused with that one request ready. The owner then performed exactly one Farcaster approval.

This fixed the timing and correlation window before the human acted. Codex never approved an identity or handled proof material; it controlled only the public interaction and evidence sequence.

## 13. The method-binding root cause

The resolver stored an injected fetch implementation or the runtime function:

```ts
this.fetcher = dependencies.fetcher ?? fetch
```

The failing path later called:

```ts
await this.fetcher(url, init)
```

In JavaScript that is a method call. The object left of the property access becomes the receiver, conceptually `fetch.call(resolverInstance, url, init)`. The deployed host implementation rejected that invented receiver before returning a `Response`.

The final repair was intentionally small:

```ts
const fetcher = this.fetcher
await fetcher(url, init)
```

Strict module code calls the local binding without the resolver receiver, preserving normal runtime `fetch()` semantics. The receiver fix was recorded in commit `23e1e77`; cumulative Alpha 0.2 work later merged normally in `d5f0748`, preserving activation ancestry.

## 14. Why arrow-function mocks hid it

Most tests injected `async () => new Response(...)`. Arrow functions do not receive dynamic `this`, so calling them as `this.fetcher(...)` does not reveal the receiver supplied by the call site. The mocks verified arguments and results but not invocation semantics.

The regression uses a normal function with an explicit `this` parameter and fails unless `this === undefined`. Host APIs, extracted methods, SDK callbacks, and injected runtime functions deserve at least one receiver-sensitive test.

## 15. Exact-runtime reproduction: value and limits

Pinned Workerd reproduction found the redirect defect because it exercised the incompatible option. It missed the receiver defect because it bypassed the resolver. The stronger standard is:

```text
same source path
+ same dependency construction
+ same invocation form
+ same runtime/configuration
+ same upstream contract
```

The final proof combined Node contract tests, receiver-sensitive coverage, pinned Workerd checks, a deployed same-path synthetic probe, and one controlled fresh browser flow. None was treated as a substitute for the others.

## 16. One cumulative PR and history-preserving merges

The repair remained on the cumulative Alpha 0.2 activation line rather than generating competing release candidates. Reviewable commits captured redirect compatibility, closed stages, classification hardening, request/body separation, removal of browser-only fetch modes, and receiverless invocation. Normal merges preserved the earlier activation and review ancestry instead of squashing the incident into one opaque commit.

That structure materially improved auditability, rollback, and later reconstruction.

## 17. Approaches that consumed time

Reasonable but insufficient approaches included:

- treating zero unhandled exceptions as success;
- using the denial panel to classify a fresh exchange;
- relying on one generic failure event;
- asking human approval to test signing, runtime networking, resolver behavior, and UI at once;
- using temporary tails as durable evidence;
- parsing multiline output line by line;
- treating a direct-fetch Workerd spike as proof of the resolver class;
- changing browser-oriented request members before isolating the failing phase;
- repeating a release ceremony for diagnostic-only changes;
- assuming a green local call disproved a production invocation defect.

Removing browser-only `cache` and `credentials` modes still improved the final Worker request, but it was not the receiver root cause.

## 18. Approaches that worked exceptionally well

The strongest controls were one-time challenge consumption, fail-closed token issuance, fixed private procedure coordinates, bounded epoch validation, closed static stages, a same-path synthetic probe, a receiver-aware normal-function test, pinned Workerd, exact-version observation, one cumulative PR, a pristine browser, Codex-controlled challenge initiation, protected aggregates before and after, and a ready rollback version.

The pattern was consistent: replace an assumption with an independently testable boundary.

## 19. Synthetic probes before human QA

Human approval should prove the irreducibly human portion: the intended account approved, the browser forwarded the completed proof, the exchange correlated with that approval, and the UI represented admission correctly. It should not be the first test of service signing, Workerd compatibility, the private procedure, or response parsing.

The production resolver probe passed before the final approval, reducing the human step to its smallest useful scope.

## 20. Protected aggregate verification

Every diagnostic and release phase checked the protected invariant:

```text
61 world tiles
0 allowlist rows
0 enabled identities
0 players
0 castles
```

It remained unchanged after probes, Worker deployments, the fresh exchange, expected denial, **CHECK AGAIN**, and remembered-session restoration. This proved both that diagnostics were read only and that successful authentication did not silently become admission.

## 21. Lessons for Cloudflare, SpacetimeDB, and Farcaster

For Cloudflare: test subrequests in pinned Workerd; execute the production abstraction; treat host functions as receiver-sensitive; express redirect rejection through supported options; and prefer durable versioned events over live tail.

For SpacetimeDB: keep authorization procedures fixed, private, bounded, and read only; validate exact types; retain module-side issuer/role/deadline/admission/epoch checks; and treat a valid but unadmitted token as expected closed-alpha behavior.

For Farcaster: isolate scarce human approval; independently verify proof in the Worker; bind it to the exact challenge/domain/URI/identity/expiry; never retain raw proof data; and keep authentication separate from admission.

## 22. Reusable debugging decision tree

```text
Verify exact source, Worker version, Pages build, module issuer,
rollback availability, and protected aggregate.
  |
  +-- Challenge failed -> inspect origin/body/rate/replay/browser transport.
  |
  +-- No correlated exchange after approval -> inspect channel polling,
  |   generation cancellation, proof completion, and challenge correlation.
  |
  +-- authorization_unavailable -> run the authenticated resolver probe.
  |     signing -> signing config, key import, claims, clock
  |     fetch_request -> Workerd invocation, receiver, RequestInit, abort
  |     fetch_body -> stream transport and deadline
  |     timeout -> determine which bounded phase stalled
  |     upstream_status -> issuer/audience/roles/procedure/redirect/health
  |     response_validation -> media type, size, JSON, integer/range
  |     success -> stop changing resolver; run one pristine browser attempt
  |
  +-- Token rejected -> browser claim validation and SDK token handoff.
  |
  +-- Database failed -> issuer/audience, bindings, protocol bootstrap.
  |
  +-- Empty admission denied -> expected; realm must remain unmounted.
  |
  `-- Unexpected state appeared -> stop as a security/integrity incident.

Always re-run exact-coordinate and protected-aggregate verification.
```

A stage narrows the search area; it does not justify inventing a more specific cause than the evidence supports.

## 23. Diagnosis, candidate, and release modes

| Mode | Purpose | Allowed work | Exit condition |
| --- | --- | --- | --- |
| Diagnosis | Classify the boundary | Read-only evidence, closed stages, exact-runtime spikes, read-only synthetics | One reproducible cause or one precise remaining stage |
| Candidate | Prove a repair | Full tests, exact candidate deployment, probe, aggregate, rollback | Candidate independently green at exact coordinates |
| Release | Converge approved state | One pristine human QA, normal merge, final deploy, production verifier, observation | Main, deployment, Worker, module, docs, and evidence agree |

Diagnosis should not repeatedly consume human approval. Candidate work is not shipped. Release mode cannot substitute source intent for deployed coordinates.

## 24. What to improve next

1. Keep durable allowlisted Worker events with exact version metadata.
2. Retain live tail only as optional supervised support.
3. Run the production resolver class in an exact Workerd integration test.
4. Maintain a signing/request/body/status/validation fault matrix.
5. Probe the resolver automatically after deploy and at low frequency.
6. Record source, Worker, Pages, module, asset, and probe coordinates together.
7. Automate the pristine-browser baseline, pausing only for human approval.
8. Keep remembered-session QA as a separate phase.
9. Replace the long-lived browser bearer with short access plus a trusted HttpOnly session before broad admission, or maintain an explicit expiring accepted-risk record.
10. Exercise signing-key, admin-secret, session-revocation, rollback, and disaster-recovery drills.
11. Maintain one incident commander, one production mutator, and read-only reviewers.
12. Preserve normal merge ancestry and reviewable incident commits.

The central lesson is simple: remain quiet to attackers, but make every internal boundary independently and safely testable for operators.
