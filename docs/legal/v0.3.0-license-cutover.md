# v0.3.0 License Cutover

This is the operational checklist for the first mechanical licensing boundary. Do not execute it in the preparation PR while the package version is `0.2.0`. Do not bump the product version merely to force this change.

The boundary is exact:

```text
v0.2.0 and earlier = legacy 0BSD + CC0-1.0 policy
v0.3.0 and later   = Apache-2.0 + CC-BY-4.0 policy for new or modified Warpkeep work
```

## Preconditions

- [ ] Create the first intentional `v0.3.0` development/release commit from the intended base.
- [ ] Confirm the release scope and provenance review with the maintainers.
- [ ] Confirm that no third-party, externally governed, font, model, audio, video, image, reference, or generated file is being relicensed without authority.
- [ ] Confirm the package version is being changed for a real product release, not only for licensing convenience.

## Exact legal texts and legacy preservation

- [ ] Replace the active root `LICENSE` with the exact, unmodified Apache-2.0 legal text from `https://www.apache.org/licenses/LICENSE-2.0.txt`.
- [ ] Preserve the historical 0BSD text byte-for-byte at an unambiguous legacy path such as `licenses/legacy/LICENSE-0BSD-v0.2.0`.
- [ ] Add the exact, unmodified CC-BY-4.0 legal code from `https://creativecommons.org/licenses/by/4.0/legalcode.txt` at an active path such as `LICENSE-CC-BY-4.0`.
- [ ] Preserve the historical CC0 text byte-for-byte at an unambiguous legacy path such as `licenses/legacy/LICENSE-CC0-1.0-v0.2.0`.
- [ ] Add a concise Apache `NOTICE` file with the factual project/contributor attribution required by the Apache policy.
- [ ] Do not delete or alter the historical legal texts; historical 0BSD and CC0 grants remain valid.

## Metadata and path mapping

- [ ] Update every package manifest's software license field to `Apache-2.0`, including the root, `services/auth-bridge`, and `spacetimedb` manifests.
- [ ] Update asset/documentation policy language to `CC-BY-4.0` for new or modified confirmed project-owned material.
- [ ] Add or update path-based SPDX/REUSE metadata, including generated binding and externally governed exceptions.
- [ ] Add SPDX identifiers to new or modified files where appropriate without editing generated headers that regeneration would overwrite.
- [ ] Record the full cutover commit SHA in [`LICENSING.md`](../../LICENSING.md) and this checklist.
- [ ] Record the release/version and exact source snapshot in `CHANGELOG.md` and `docs/releases/`.

## Provenance and generated output

- [ ] Re-audit `ASSETS-LICENSE.md`, `docs/reference/**`, manifests, runtime derivatives, and third-party notices.
- [ ] Confirm rights for every creative file receiving CC-BY-4.0; keep uncertain and externally governed files under their original terms.
- [ ] Run the pinned SpacetimeDB binding generator and verify output with `npm run stdb:verify-bindings`.
- [ ] Confirm generated bindings are covered only where Warpkeep has the right to license the generated output; retain upstream generator/SDK notices.
- [ ] Confirm dependencies, vendored code, fonts, models, audio, video, images, hosted services, and GitHub Actions retain their upstream licenses.

## Validation and release

- [ ] Run `npm ci`.
- [ ] Run `npm run verify:licenses`.
- [ ] Run `npm test`.
- [ ] Run `npm run typecheck`.
- [ ] Run `npm run build`.
- [ ] Run `GITHUB_PAGES=true npm run build`.
- [ ] Run `GITHUB_PAGES=true DEPLOY_BASE=/ npm run build`.
- [ ] Run `npm audit`.
- [ ] Run `git diff --check` and inspect the complete staged file list.
- [ ] Run bridge checks, module verification, and generated-binding verification when those projects are included in the release.
- [ ] Validate README/legal-document links and confirm no secret, proof, private key, or token entered the change.
- [ ] Confirm the source snapshot, active license files, package metadata, path mappings, and build stamp all agree.
- [ ] Create the annotated `v0.3.0` tag and GitHub Release only after the matching merge commit is deployed and verified.
