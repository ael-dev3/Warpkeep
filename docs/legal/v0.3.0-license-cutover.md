# v0.3.0 License Cutover

This records the completed operational checklist for Warpkeep's first mechanical licensing boundary. Commit A `3ff1775c11f780ec9d122d05483405bf034fcdfe` established the complete v0.3.0 policy snapshot; direct child Commit B `6caca9d40ba51061f36f074848a3db7ff7f955d4` added the machine-readable attestation. Both commits are immutable ancestors of the release line.

The boundary is exact:

```text
v0.2.0 and earlier = legacy 0BSD + CC0-1.0 policy
v0.3.0 and later   = Apache-2.0 + CC-BY-4.0 policy for new or modified Warpkeep work
```

The verifier accepts only complete SemVer 2.0.0 values. A valid version whose core is `0.3.0` or later, including `0.3.0` prerelease/build forms, requires the completed cutover. This prevents a prerelease suffix from bypassing the policy boundary.

## Two-commit protocol

The boundary is deliberately split so the attestation is mechanically possible:

1. **Commit A — cutover:** perform the complete legal-text, package-metadata, historical-license, NOTICE, REUSE/path-map, inventory, and release-evidence changes. Commit A must have one pre-v0.3.0 historical parent and must not contain the machine-readable cutover record.
2. **Commit B — attestation:** add `docs/legal/v0.3.0-cutover.json` with Commit A's full lowercase SHA. Run the verifier at Commit B; it proves that A exists, is an ancestor, represents the v0.3.0 boundary, and still agrees with current `HEAD`.

Create A and B on the intended final base. Once B records A, do not amend, rebase, squash, cherry-pick, or otherwise rewrite either commit. Merge them with a commit-preserving strategy. If their base must change, recreate and re-attest the pair instead of carrying a stale SHA forward.

## Cutover record

Commit B added `docs/legal/v0.3.0-cutover.json` with this exact shape and Commit A's lowercase 40-character SHA:

```json
{
  "schemaVersion": 1,
  "cutoverVersion": "0.3.0",
  "cutoverCommitSha": "3ff1775c11f780ec9d122d05483405bf034fcdfe"
}
```

The verifier reads the JSON record rather than a SHA or tag in prose. `npm run verify:licenses` resolves that object as a commit, requires it to precede `HEAD`, inspects its complete tracked snapshot with `git show`, enumerates every tracked package manifest, and rechecks the active state at `HEAD`.
